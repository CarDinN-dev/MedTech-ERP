# Incident Response Playbook

Local demo scope: browser localStorage data, demo authentication, local imports, generated PDFs, and simulated ERP workflows. Do not enter production secrets or patient data into this demo.

## First 15 Minutes

1. Preserve evidence: export the audit log and export all local demo data from Administration -> Local Data Tools.
2. Stop the risky action: sign out affected users, reset demo data only after exporting evidence, and avoid re-importing suspicious files.
3. Record the timeline: event time, user, role, module, record ID, action, severity, result, and screenshots where useful.
4. Classify severity:
   - info: normal activity.
   - warning: approval, reset, archive, backup, or unusual workflow action.
   - high: failed login, permission denied, failed restore, suspicious import.
   - critical: reset all data, restore backup, exposed secret, confirmed tampering.

## Suspicious Login

Indicators: repeated login failures, unexpected user, unknown role, login outside expected demo activity.

Response:
1. Filter Audit log by module `Authentication`, action `LOGIN`, result `failure`, severity `high`.
2. Capture user/email, timestamps, browser, and source IP shown in the demo audit row.
3. Reset the affected demo user password from Administration -> Users.
4. Export audit evidence before resetting any demo data.

## Unauthorized Approval

Indicators: approval by requester, wrong role, unexpected PO/payroll/quotation/costing approval.

Response:
1. Filter by actions `APPROVE`, `REJECT`, `SUBMIT APPROVAL`, and module `Approvals`.
2. Compare user/role with approval request fields and segregation warnings.
3. Export affected module rows and approval matrix rows.
4. Revert by cancelling/rejecting the approval or restoring a known-good backup.

## Data Tampering

Indicators: unexpected create/update/delete/archive, changed amounts, missing records, unauthorized reset.

Response:
1. Filter Audit log by record ID or module and review Before/After summaries.
2. Export current local data before applying any fix.
3. Restore from a validated backup if records are corrupted.
4. Reset only the affected module when a full reset is unnecessary.

## Exposed Secret

Indicators: password, token, API key, cookie, or authorization value pasted into a record, note, file, or import.

Response:
1. Treat as critical even in local demo mode.
2. Remove the exposed value from local records and browser storage.
3. Rotate the real secret outside this demo if it was real.
4. Export audit evidence after removal; audit sanitization redacts sensitive keywords, but screenshots or files may still contain them.

## Corrupted Local Data

Indicators: app falls back to seed data, import restore fails, records vanish after bad localStorage edits.

Response:
1. Export current local data if the app still loads.
2. Try importing the latest known-good backup JSON from Local Data Tools.
3. If one module is broken, reset one module first.
4. Use Reset All Demo Data only after evidence and backups are no longer needed.

## Vulnerable Dependency

Indicators: `npm audit` reports moderate or higher issues, CI security check fails.

Response:
1. Run `npm.cmd run security:audit`.
2. Record package, severity, vulnerable range, and fix version.
3. Prefer `npm.cmd update <package>` or an existing override; avoid adding new packages.
4. Run typecheck, tests, and build after dependency changes.

## Malicious File Or Import Attempt

Indicators: wrong extension, oversized file, malformed JSON, import formula injection, unexpected backup keys.

Response:
1. Do not retry the same file after validation fails.
2. Keep the failed file name and error message from the audit row.
3. Export current local data before any restore attempt.
4. Use only JSON backups exported by the app and Excel/CSV templates generated by the app.

## Evidence Collection

Collect:
1. Audit log export with filters cleared.
2. Backup JSON exported before recovery.
3. Screenshots of affected module rows.
4. Browser console error text, if any.
5. Exact file name and size for suspicious imports.
6. Commands run and results, including typecheck/build/security audit.

## Recovery Steps

1. Export evidence.
2. Choose the smallest recovery: undo workflow, reset one module, restore backup, or reset all demo data.
3. Validate the app loads and affected module records are readable.
4. Confirm Audit log contains the recovery action.
5. Run `npm.cmd run typecheck` and `npm.cmd run build` after code or dependency changes.
